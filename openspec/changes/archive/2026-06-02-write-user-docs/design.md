## Context

`focus` has a well-developed `README.md` and internal planning documents in `docs/`, but no standalone user-facing documentation. The `docs/` directory currently mixes internal artifacts (plan, roadmap, research notes) with the design doc. Before release the directory needs a clean separation so users see reference material and contributors see planning history.

The README already covers the essential quickstart flow, but it's dense and doesn't have room for full reference content. Separate doc files let each topic breathe.

## Goals / Non-Goals

**Goals:**
- Clean `docs/` top level with only user-facing files
- Getting started guide with installation placeholder (distribution TBD)
- Configuration reference covering both `.focus.yaml` and `~/.config/focus/config.yaml`
- Profile system reference (built-in catalog + custom profile format)
- Brief focus vs. devcontainers comparison
- About page with OpenSpec/SDD portfolio note
- Casual but competent tone throughout

**Non-Goals:**
- Autogenerated docs from source
- Versioned docs site
- Changing the README significantly (it's already solid)
- Documenting internal architecture (that stays in `docs/internal/design.md`)

## Decisions

**Separate files per topic, not a single doc**
A monolithic doc is harder to link to, harder to navigate, and grows unwieldy. Separate files let users go directly to what they need.

**`docs/internal/` for planning artifacts**
Alternatives: delete them, leave them in place, move to a top-level `planning/` dir. `docs/internal/` keeps them discoverable for contributors without cluttering the user-facing surface. The `design.md` architectural reference also moves here — it's valuable for contributors but not end users.

**Installation section uses TBD placeholder**
Distribution method (Homebrew, npm, etc.) is undecided. The doc describes what prerequisites to install and holds a clearly-labeled placeholder for the install command. This is honest and avoids premature commitment.

**`docs/getting-started.md` mirrors the README quickstart but adds depth**
The README quickstart stays as-is. `getting-started.md` expands on it: prerequisites per platform, the install placeholder, first-run walkthrough, and a note on what happens inside the container.

## Risks / Trade-offs

- Docs written before release will reflect the current feature set; any features added later need doc updates → Mitigated by keeping docs close to source in the same repo
- The focus vs. devcontainers section could read as defensive → Keep it factual and brief; one paragraph max
- Moving internal docs breaks any hardcoded links → README links to `docs/design.md` which will move; update that link

## Open Questions

- Should `docs/design.md` stay accessible from the README or is it purely internal? → Treat as internal for now; no prominent link needed
